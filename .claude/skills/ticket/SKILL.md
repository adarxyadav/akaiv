# Ticket

Route arriving work into the project's tickets file before anyone builds — no ticket, no work, and this skill is how work earns its ticket.

1. **Write the ticket.** One checkbox per unit of work, in the project's tickets file.
2. **Blind-spot pass.** Unfamiliar territory → list the open questions under the ticket, then sort each one. A **fact** already lives somewhere — code, schema, docs, git history — so dig it up now and record what you found. A **decision** exists nowhere yet: it goes to the human, one at a time, each with your recommended answer, and the ticket waits for their call.
3. **Name the check.** Write under the ticket the observable event that will close it — a passing suite, a deployed page, the rule-7 quiz for judgment work — owned by someone or something other than whoever will do the work. Climb the hand-off ladder only as far as the ticket can truly name: check, then stop condition, then trigger.

Done when the ticket sits in the file with every question sorted, every decision answered by the human, and the check named — while the building itself has yet to begin. Starting it is /work's job, on the human's go.
